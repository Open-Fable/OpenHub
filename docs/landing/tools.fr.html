Outils

Cinq outils. Une fenetre. Tout en commun.

Deux sont integres a OpenHub : le chat et l'orchestrateur. Trois sont des outils open-source matures, OpenCode, Open Design et OpenWork, embarques sans toucher une ligne de leur source. Quel que soit celui que tu ouvres, il utilise les memes cles API, le meme theme, la meme memoire et le meme contexte projet.

Integre a OpenHub

Chat

Discute avec n'importe quel modele, sans ouvrir une deuxieme app.

Un vrai chat integre a la sidebar. Change de modele en cours de conversation, attache des images et des fichiers, active la recherche web quand tu as besoin de faits frais, et tire les instructions d'un projet dans le fil. Chaque conversation est sauvegardee et stockee sur le disque.

  • N'importe quel modele. Anthropic, OpenAI, OpenRouter, Google Gemini ou Ollama local, choisi dans un menu sous la zone de saisie.
  • Effort de raisonnement. Pour les modeles qui le supportent, regle l'intensite de reflexion, memorisee par modele.
  • Pieces jointes. Glisse des images (PNG, JPEG, GIF, WebP) ou des fichiers texte.
  • Recherche web. Active-la et le proxy enrichit le contexte avec des resultats en direct.
  • Contexte projet. Lie un projet pour injecter ses instructions dans la conversation.
  • Un historique a toi. Sauvegarde auto, sauvegarde sur disque, et epinglable en haut de la liste.

Orchestrateur

Un objectif en entree. Une equipe d'agents s'en charge. Un dossier de vrais fichiers en sortie.

Decris ce que tu veux en langage courant, « un site de tarifs avec demo de paiement », « une librairie Python avec des tests », « un rapport d'etude de marche ». OpenHub le decoupe en un graphe d'agents specialises, les execute dans le bon ordre, et ecrit le resultat dans un dossier sur le disque. Un vrai livrable. Pas une transcription.

Ensuite, un controle qualite deterministe entre en jeu — du code pur, aucun modele. Il analyse statiquement le resultat : imports casses, fichiers manquants, liens morts, pages non stylees, et API qui ne s'alignent pas entre les fichiers de deux agents. Les defauts forcent un echec quoi que dise le modele verificateur. Seul l'agent qui possede le fichier casse est relance.

Lire la reference du moteur
  • Des agents specialises. Recherche, work (contenu & design system), design (maquettes), code, et un verificateur en lecture seule, chacun avec ses regles.
  • Il se planifie tout seul. Une boucle de planification decoupe l'objectif, cree des sous-agents, et declare un contrat de fichiers pour chacun.
  • Graphe de dependances. Les agents declarent leurs dependances ; les resultats amont alimentent l'aval comme contexte.
  • Le modele ne se note pas lui-meme. Les defauts structurels sont detectes par analyse statique et forcent un echec, meme si le verificateur dit que c'est bon.
  • Auto-correction. Quand le controle echoue, seul l'agent responsable est relance, puis le travail est re-audite.
  • Tu peux le piloter. Donne ton retour sur un run termine et il repart dans le cycle correctif avec tes notes.
OpenCode · par SST · embarque sans modification

Code

Un agent de code de niveau terminal, dans sa propre vue isolee.

OpenCode est un agent de code serieux. Il lit et ecrit des fichiers dans ton projet et execute des commandes pour faire le travail. Dans OpenHub il tourne comme un serveur local dans une vue isolee, theme pour coller au reste de l'app, configure par le meme proxy que tout le reste.

C'est aussi le moteur en dessous. Les trois outils integres pilotent OpenCode. C'est pour ca qu'un seul fichier de config les regle tous a la fois.

  • Lit & ecrit des fichiers. Travaille directement sur ton projet sur le disque, pas sur des bouts colles.
  • Execute des commandes. Pilote la chaine d'outils pour builder, tester et verifier, dans sa propre vue.
  • Memes cles, memes modeles. Route par le proxy partage, il herite de ce que tu as configure une fois.
  • L'etat est preserve. Passe a un autre outil et reviens ; ta session est exactement la ou tu l'as laissee.
Open Design · par nexu-io · embarque sans modification

Design

Genere et affine des maquettes, sans jamais voir un terminal.

Open Design produit des maquettes d'interface en HTML, CSS et SVG et les affine au fil des iterations. Normalement il tourne comme un daemon en ligne de commande. OpenHub le lance pour toi, le capture, et le met derriere une vue graphique, pour qu'un designer qui n'ouvre jamais de terminal puisse l'utiliser comme n'importe quelle app.

Dans une orchestration, l'agent design recoit le design system de l'agent work et le transforme en maquettes fideles, que l'agent code reproduit ensuite.

  • Maquettes HTML, CSS & SVG. Du vrai markup que tu peux passer directement a un developpeur.
  • Affinage iteratif. Relance pour corriger un resultat trop mince ou rempli de placeholders jusqu'a ce que la maquette tienne.
  • Aucune ligne de commande. L'app demarre le daemon et le gere. Tu decris juste l'ecran.
  • Alimente l'orchestrateur. Les maquettes sont ecrites sur le disque et passees a l'agent code pour les construire pour de vrai.
OpenWork · par different-ai · embarque sans modification

Work

Travail structure et documents, relies a tes projets.

OpenWork gere le cote structure de la creation : contenu, documents, travail organise qui n'est ni du code ni des pixels. Dans OpenHub il partage les memes projets que tout le reste, donc les instructions et le contexte que tu definis une fois te suivent dans Work et en ressortent.

Dans une orchestration il tient le role de moteur de contenu, et il possede le design system : palette, typographie, et le look que le designer et le codeur doivent ensuite respecter.

  • Documents & donnees structurees. Redaction et sorties organisees en Markdown, JSON et CSV.
  • Projets partages. Les memes instructions de projet utilisees par Chat et l'orchestrateur s'appliquent aussi ici.
  • Definit le design system. Comme agent d'orchestration, Work fixe les couleurs et la typo que les autres doivent suivre.
  • Recherche web native. OpenHub ajoute la recherche a Work via le proxy, pas via une cle cloud integree.

Memoire

Arrete de re-expliquer qui tu es et ce que tu construis.

La memoire, c'est un profil plus un ensemble de faits courts et etiquetes. Le profil te decrit, ton role, ta stack, tes preferences. Les faits sont des details reutilisables. A chaque requete, seuls les faits pertinents sont injectes, dans un budget de tokens, pour que le prompt reste leger au lieu de trainer tout ton historique.

Tu gardes le controle. Un interrupteur global coupe entierement la memoire, et chaque projet peut s'en exclure de son cote.

  • Profil. Texte libre sur toi, injecte pour personnaliser les reponses.
  • Faits etiquetes. Notes courtes et reutilisables ; seules les pertinentes entrent dans une requete donnee.
  • Extraction auto. Laisse au choix un modele extraire de nouveaux faits de tes conversations.
  • Coupee quand tu veux. Un interrupteur global, plus une exclusion par projet.
Le liant

Les projets relient tout.

Un projet regroupe des instructions, un type, un modele prefere et un contexte. C'est ainsi qu'une seule config atteint chaque outil.

Projets autonomes

Lies a un seul outil : Code, Design ou Work. Ouvre l'outil et les instructions du projet sont injectees dans le contexte IA. Un projet autonome peut aussi devenir une brique dans une orchestration.

Projets orchestrateur

Ils ne tournent pas seuls : ils coordonnent plusieurs agents. Ils definissent l'objectif global, les agents a enchainer, et des reglages comme l'auto-distribution, le controle de coherence et la relance sur erreur.
